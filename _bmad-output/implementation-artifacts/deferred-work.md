## Deferred from: code review of story 1-6 (2026-04-28; updated 2026-05-11)

- **`crypto.randomUUID()` requires SecureContext** — `useApi` generates request-IDs via `crypto.randomUUID()`, which is only available on HTTPS or `localhost`. A developer hitting Vite's `host: '0.0.0.0'` mapping from another machine on the LAN (`http://192.168.x.y:5173/`) sees every API call throw `TypeError: crypto.randomUUID is not a function`. Add a fallback path that uses `crypto.getRandomValues` or imports a tiny v4 polyfill. **Landing point: Story 1.9 (Dockerfile / dev TLS)** — TLS termination on the dev container removes the SecureContext gap and obviates the polyfill for the local workflow. If cross-machine LAN testing on plain HTTP becomes a routine workflow before 1.9, fold the polyfill into a Story 1.6.1 patch.
- ~~**`AbortSignal` not threaded into `fetchCsrfToken`**~~ — **Resolved 2026-05-11** as part of the story 1-6 review fix-up. `fetchCsrfToken` now accepts the parent request's signal and the bootstrap fetch is aborted when the caller aborts. Tested in `useApi.spec.ts::useApi — AbortSignal threading`.
- ~~**`buildUrl` `startsWith(API_PREFIX)` admits malformed paths**~~ — **Resolved 2026-05-11.** `buildUrl` now rejects protocol-relative paths (`//...`), missing-slash typos (`/api/v1projects`), and non-string inputs at the wrapper boundary. Tested in `useApi.spec.ts::useApi — buildUrl input hardening`.
- **CSRF bootstrap fires on respondent submission too** — `useApi` requests a CSRF token on every non-GET regardless of route. Respondent submission (Story 4-3) is `public_endpoint` (CSRF-exempt), so the token is sent but unused — wasted ~50–100 ms round-trip on the respondent's first submit. Add an exception list for known-public endpoint patterns, or move CSRF bootstrap behind an explicit `pm`-only opt-in. **Landing point: Story 4-3** — that story implements `POST /api/v1/polls/{uuid}/submissions` and is the first concrete consumer; the public-endpoint allowlist lands alongside it as a one-line change.
- **`kano/__init__.py` eager imports drag flask/sqlalchemy into every submodule import** *(was listed under story 1-5; expanded here)* — the story 1-5 purity test had to fall back to AST-static inspection because `python -c "import kano.services.kano_matrix"` correctly reports flask+sqlalchemy as loaded. **Landing point: Story 7.2 (Caddy / deploy hardening)** — that story already restructures the app entry-point for production startup, the lazy-import refactor folds into it.
