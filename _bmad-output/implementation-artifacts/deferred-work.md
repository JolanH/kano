## Deferred from: review of "deterministic poll UID + link buttons" (2026-06-02)

- **Same-epoch soft-delete of a pinned poll's last feature leaves an empty, still-served poll** — once a poll exists for `(project, epoch=N)`, `poll_service.create_poll` returns it on repeat **without** re-checking active-feature count (intentional: a pinned snapshot is treated as immutable). But feature soft-delete does NOT always bump the epoch (see `test_create_poll_after_soft_delete_returns_422`: deleting the sole feature keeps `current_epoch=1` with zero active features). So the sequence create-poll → soft-delete-last-feature leaves a poll whose `_active_features_query` now resolves to zero rows; `get_poll_public` would serve a feature-less poll, and re-POST returns it as 200 instead of the old 422. This is fundamentally a pre-existing soft-delete-semantics gap (the poll breaks at delete time regardless of the re-POST), surfaced by the idempotency change. **Fix options:** block soft-deleting the last active feature of an epoch that has a poll, or have `get_poll_public` 410/404 when the pinned snapshot has zero active features. Low severity; revisit alongside any feature-deletion hardening.
- **"Copy poll URL to clipboard" can fail on Firefox/Safari** — `ProjectDetail.vue` `onCopyPollUrl` awaits `ensurePollUrl()` (a network get-or-create) before `navigator.clipboard.writeText`. On Firefox/Safari the transient user-activation needed for clipboard writes is consumed across the `await`, so the first copy (when the poll isn't cached) rejects and the PM sees the graceful "couldn't copy automatically" snackbar. Works on Chromium (the internal-tool target). **Fix options:** eagerly ensure the deterministic poll exists on page mount so the URL is ready synchronously, or expose the URL in a readonly field with an execCommand fallback like `PollSharePanel`. Medium severity on non-Chromium browsers; gracefully handled today.
